# Draft：人员详情长图装修

## Requirements（confirmed）
- 目标页面：`packages/pages/staff_detail/staff_detail`
- 目标能力：增加类似淘宝的长图详情内容
- 装修诉求：支持自主装修，可放多张无间距图片、文字等内容
- 影响范围：需要同时实现后台
- 影响范围：需要实现 uniapp 端“服务人员中心”内的简单修改
- 明确排除：本次修改不涉及 PC 端
- 后台装修方式：图文模块化
- 服务人员中心范围：只改“长图详情”这块内容
- 首版模块范围：图片模块 + 文本模块
- 空数据前台策略：为空则隐藏长图详情区域
- 文本模块能力：基础样式
- 服务人员中心编辑深度：支持增删排序图片/文本模块
- 图片上传方式：沿用现有上传链路
- staff_detail 页面承载方式：替换现有详情区
- 旧数据策略：不迁移旧数据，未配置则隐藏

## Technical Decisions
- 倾向方案：采用模块化图文数据结构，而非单纯富文本
- 已确认：服务人员中心只针对“长图详情”做简单修改，不扩展到其他资料项
- 首版遵循 YAGNI：只提供“图片、文本”两类模块，不预先扩展更多复杂区块
- 展示兜底策略：未配置时直接隐藏，不增加额外占位
- 文本模块暂按“基础样式”规划，不做完整富文本
- 服务人员中心不只查看，需具备模块增删与排序能力
- 数据持久化优先采用 JSON 区块结构，降低 uniapp / 小程序富文本兼容与性能风险
- 图片上传沿用现有上传组件、资源库与上传接口，不为本次功能单独发明链路
- 前台展示策略：长图详情替换现有详情区，而不是额外叠加新区域
- 历史兼容策略：不做旧字段迁移或兼容映射，降低本次改造范围与风险

## Research Findings
- 已启动代码库调研：人员详情页、后台编辑页、后端接口与相似装修能力
- 已启动外部调研：uni-app / 微信小程序长图图文详情的生产实践与限制
- 已启动测试基础设施评估：admin / uniapp / server / pc
- 外部调研结论：长期存储更推荐结构化 JSON 区块，而不是直接长期存 HTML 字符串
- 外部调研结论：长内容渲染需关注图片懒加载、节点规模和滚动性能
- 外部调研结论：图片上传工作流宜复用现有上传链路，最终在区块数据中保存图片 URL
