# 项目功能审计报告（`uniapp` + `admin` + `server`）

更新时间：2026-04-12

## 一、审计范围与判定口径

本次只审计以下三端：

- `uniapp`：用户主业务端、服务人员端、移动端管理员看板入口
- `admin`：后台运营与财务管理端
- `server`：实际业务能力与数据模型来源

本报告不看“有没有文件”，只看“有没有形成正确业务闭环”。

每个业务域按以下口径判断：

- `功能状态`：必用、废弃/下线、不完整、缺失
- `完整度`：0-3 分
- `流程正确性`：0-3 分

评分定义：

- `完整度`
  - `3`：主流程与主要管理入口齐全
  - `2`：主流程可走，但有明显缺口
  - `1`：有入口但残缺明显
  - `0`：仅壳子/占位/未形成能力
- `流程正确性`
  - `3`：入口、状态流转、异常分支、后台处理基本正确
  - `2`：主流程基本对，但异常分支或端间联动有缺口
  - `1`：主流程存在明显断点或错误
  - `0`：流程错误或根本未形成闭环

## 二、总体结论

### 1. 当前项目的真实核心，不是“商城”，而是“婚礼服务预约系统”

当前代码的主业务闭环是：

- 人员展示与预约
- 档期查询
- 订单创建与支付
- 订单履约与变更
- 售后与评价
- 服务人员中心
- 后台运营与财务管理

### 2. 当前整体成熟度判断

- `uniapp` 主交易与履约能力：`完整度 2/3`，`流程正确性 1-2/3`
- `admin` 运营后台能力：`完整度 2/3`，`流程正确性 2/3`
- `server` 后端能力：`完整度 3/3`，但存在“后端已有，前端未接完”的不对称问题

### 3. 当前最重要的结论

- 当前必须保住的功能已经很明确，但还存在少量 `P0/P1` 级流程问题。
- 最大问题不是“没有功能”，而是：
  - 交易主链路有关键步骤未真正闭环
  - 若干后台/后端能力没有前台入口
  - 少量后台入口是旧路径或占位页
  - 装修系统部分组件仍是 mock 演示态，不是真数据态

## 五、不完整或流程不正确的功能

## P0

### 1. 预约下单主链路没有真正使用“档期锁定”能力

结论：

- `功能状态`：不完整
- `完整度`：2/3
- `流程正确性`：1/3

证据：

- 前端下单确认页只调用了 `previewOrder` 与 `createOrder`
  - `uniapp/src/packages/pages/order_confirm/order_confirm.vue`
- 人员详情页只调用了 `checkScheduleAvailable`
  - `uniapp/src/packages/pages/staff_detail/staff_detail.vue`
- 前端未发现对 `lockSchedule` / `releaseLock` 的实际调用
  - 搜索结果仅存在 API 定义：`uniapp/src/api/schedule.ts`
- 后端明确提供了锁定/释放能力
  - `server/app/api/controller/ScheduleController.php`

风险：

- 用户看到“可约”并不代表该时间段已被当前订单占住
- 高并发或多人同时预约时，存在被抢占、重复下单、后续校验失败的风险
- 这会直接影响交易主链路的“流程正确性”

建议：

- 把“选档期 -> 锁档 -> 预览 -> 建单 -> 成功后持有/失败释放”补成真实流程
- 未补之前，不应把当前预约闭环评为完整

### 2. 后台工作台存在错误或高风险跳转

结论：

- `功能状态`：不完整
- `完整度`：2/3
- `流程正确性`：1/3

证据：

- `admin/src/views/workbench/index.vue:289` 把“待审核员工”跳到 `/service/staff`
- 当前仓库未看到与 `/service/staff` 对应的实际页面字符串引用，仅看到旧兜底 `'/service/staff/edit'`
  - `admin/src/views/staff/lists/index.vue:153`

风险：

- 工作台是高频入口，错误跳转会直接破坏后台办事效率
- 这类问题说明后台路由命名仍夹杂旧路径

建议：

- 排查工作台全部快捷入口与待办跳转
- 用当前真实菜单路由统一替换旧路径

- 演示组件要么下线，要么接入真实数据源

### 4. 评价体系有后端“申诉”能力，但用户端未形成真实入口

结论：

- `功能状态`：不完整
- `完整度`：1/3
- `流程正确性`：1/3

证据：

- 用户端 API 已提供 `submitAppeal`
  - `uniapp/src/api/review.ts`

  - `server/app/api/controller/ReviewController.php`
- 但在 `uniapp` 页面中未发现实际调用入口

风险：

- 后台已有 `评价申诉` 管理，但用户侧无法发起真实申诉
- 形成单边能力

建议：

- 如果申诉是正式业务，就在评价详情或订单评价页补入口
- 如果不是正式业务，就从后台能力规划中降级

### 5. 发票能力只存在后台/后端，用户侧缺完整申请闭环

结论：

- `功能状态`：不完整
- `完整度`：1/3
- `流程正确性`：1/3

证据：

  - `server/app/common/model/financial/Invoice.php`
  - `server/app/adminapi/logic/financial/InvoiceLogic.php`

  - `admin/src/views/financial/invoice/index.vue`
- 用户端未发现发票申请 API、页面、提交流程

- 当前更像“后台登记与处理能力”，不是“用户申请发票闭环”
- 如果产品定位里包含发票服务，这就是缺口

建议：

- 明确发票是否属于正式业务
- 如果属于正式业务，补用户端申请入口与订单侧关联

### 6. 服务人员证书能力存在于后端和详情页展示，但后台缺明显管理入口

风险：

- 数据可展示，但后台缺直观运营入口

建议：

## 九、最终判断

如果只用一句话概括当前项目现状：

这个项目的核心业务能力已经成型，主框架不是缺功能，而是“少量关键流程未闭环、若干能力前后台不对称、部分旧页面和装修组件还停留在过渡态”。

如果按经营优先级排序，当前最需要看的不是再加新功能，而是先把这三件事做对：

- 交易主链路的锁档闭环
- 后台高频入口的准确跳转
- 已有后端能力与用户端入口的补齐策略
