# 消息推送链路审计结论

## 1. 检查范围与判断口径

- 检查范围：站内消息、微信小程序订阅消息、服务人员相关的企业微信内部消息推送。
- 检查方式：以静态代码审计为主，辅以无副作用验证，不修改任何业务代码或配置。
- 判断口径：
  - “流程完善”指业务触发、发送能力、配置入口、前端接入四部分闭环完整。
  - “可以跑通”指按当前仓库代码与默认初始化数据，链路具备落地执行条件；不把“理论上支持、但默认未配置”的情况视为已跑通。
  - 微信订阅消息、企业微信消息都依赖外部平台真实配置；仓库内若未体现这些配置，不按“已跑通”判定。

## 2. 总体结论

| 链路 | 完整度判断 | 当前能否跑通 | 结论 |
| --- | --- | --- | --- |
| 站内消息 | 基本完整 | 基本可以 | 主链路已形成闭环，代码层面可认为基本可跑。 |
| 微信订阅消息 | 不完整 | 不能按默认状态认定为可跑通 | 发送器、授权逻辑、场景模型都已存在，但默认模板绑定、前端授权入口、业务触发接入均未闭环。 |
| 企业微信内部消息 | 不完整 | 不能按默认状态认定为可跑通 | 发送器结构基本成立，但后台配置入口缺失企微关键字段，顾问成员维护闭环不完整。 |

## 3. 站内消息链路

### 3.1 当前流程

站内消息的主流程已经比较清晰：

1. 业务侧通过 `StationNotificationService::send()` / `sendUnique()` 写入站内通知。  
   参考：`server/app/common/service/StationNotificationService.php:32-104`
2. 通知实体由 `Notification::send()` 落库。  
   参考：`server/app/common/model/notification/Notification.php:55`
3. API 侧通过 `NotificationLogic::lists()`、`detail()` 提供消息列表与详情。  
   参考：`server/app/api/logic/NotificationLogic.php:25-75`
4. UniApp 端消息中心页面已接入消息列表、详情、已读、清空已读、删除等交互。  
   参考：`uniapp/src/packages/pages/notification/index.vue:108-359`

### 3.2 审计判断

- 主链路是闭环的，消息生产、读取、展示都已接上。
- 站内消息不依赖外部平台模板或第三方凭证，仓库内即可形成自洽链路。
- 从代码结构看，站内消息是当前三条链路里最完整的一条。

### 3.3 结论

- 站内消息：**代码层面基本可跑通**。
- 当前未发现阻断主流程的明显缺口。

## 4. 微信小程序订阅消息链路

### 4.1 当前流程

订阅消息链路已经具备基础骨架：

1. 后端维护订阅场景、模板、用户授权记录，并按场景查询可授权项。  
   参考：`server/app/api/logic/SubscribeLogic.php:36-223`
2. 小程序端提供订单场景与售后场景订阅授权函数。  
   参考：`uniapp/src/utils/subscribe.ts:229-303`
3. 订单确认页在提交流程中接入了订单类订阅授权。  
   参考：`uniapp/src/packages/pages/order_confirm/order_confirm.vue:324,494,599-611`
4. 后端发送器已经覆盖订单提醒、退款结果、工单更新等场景。  
   参考：`server/app/common/service/SubscribeMessageService.php:412-456`

### 4.2 阻塞项

#### 🔴 阻塞项 1：默认初始化场景没有把场景和模板真正绑定起来

- `la_subscribe_message_scene` 的初始化数据虽然创建了场景，但场景表中的 `template_id` 默认未形成真实可用绑定，后端在过滤可订阅场景时会直接跳过未绑定模板的场景。  
  参考：`server/app/api/logic/SubscribeLogic.php:162-223`
- SQL 中模板表写入的是占位值，例如 `TEMPLATE_ID_ORDER_CREATE`、`TEMPLATE_ID_REFUND_RESULT` 等，备注也明确写了“需在微信后台申请模板后更新 template_id”。  
  参考：`server/sql/2.0.0.20260201/update.sql:2074-2082`

**影响：**

- 前端拿不到完整的可授权模板列表。
- 即使后端存在发送器，没有真实模板 ID 也无法按默认仓库状态发送成功。

#### 🔴 阻塞项 2：前端授权入口只覆盖订单类场景，售后类未接入实际业务流程

- 小程序工具层提供了两组函数：
  - `subscribeOrderScenes()`：订单创建、确认、支付。  
    参考：`uniapp/src/utils/subscribe.ts:229-254`
  - `subscribeAfterSaleScenes()`：工单更新、退款结果。  
    参考：`uniapp/src/utils/subscribe.ts:256-281`
- 当前页面侧明确接入的只有订单类授权。  
  参考：`uniapp/src/packages/pages/order_confirm/order_confirm.vue:324,609`
- 未发现售后流程页面对 `subscribeAfterSaleScenes()` 的实际调用。

**影响：**

- 工单更新、退款结果等后续场景，即使后端发送器存在，也缺少前端授权闭环。

#### 🔴 阻塞项 3：部分发送器存在，但未发现明确业务触发点

- 退款结果与工单更新发送器已经实现：  
  参考：`server/app/common/service/SubscribeMessageService.php:434-456`
- 但当前仓库中未确认到与退款审核完成、工单状态变更强绑定的统一触发入口。

**影响：**

- 发送能力存在，不等于真实业务一定会发。
- 当前更接近“能力预埋”，而不是“稳定跑通”。

### 4.3 审计判断

- 订阅消息链路目前属于“后端能力已搭框架、但默认配置和业务接入不完整”的状态。
- 这条链路最关键的问题不是发送 SDK，而是：
  - 模板 ID 没有真实配置；
  - 场景与模板绑定未在默认数据中闭环；
  - 前端授权入口覆盖不全；
  - 部分业务事件缺少明确触发接入。

### 4.4 结论

- 微信订阅消息：**当前不能认定为已完整跑通**。
- 如果线上环境已经人工补齐真实模板 ID，并在售后页补了授权入口，再把业务事件触发补齐，链路才可能转为可跑状态；但这些条件当前仓库内没有闭环证据。

## 5. 企业微信内部消息链路

### 5.1 当前流程

企业微信内部消息的发送器和顾问通知尝试已经存在：

1. `WeComMessageService` 会从 `customer_service` 配置组读取 `wecom_enabled`、`wecom_corp_id`、`wecom_secret`、`wecom_agent_id`。  
   参考：`server/app/common/service/WeComMessageService.php:116-119`
2. 发送对象依赖服务人员或顾问维护的 `wecom_userid`。  
   参考：`server/app/common/service/WeComMessageService.php:34-35,51-52`
3. 咨询分配后，业务逻辑会尝试调用企微通知。  
   参考：`server/app/api/logic/CustomerServiceLogic.php:21,171`
4. 服务人员编辑页已有“企微成员 ID”字段。  
   参考：`admin/src/views/staff/lists/edit.vue:66-68,687`

### 5.2 阻塞项

#### 🔴 阻塞项 1：后台客服设置缺少企微关键配置入口

- `CustomerServiceLogic` 当前只处理以下配置项：`qr_code`、`wechat`、`phone`、`service_time`、`contact_link`、`tips`。  
  参考：`server/app/adminapi/logic/setting/CustomerServiceLogic.php:36-63`
- 未看到对以下企微关键配置的维护入口：
  - `wecom_enabled`
  - `wecom_corp_id`
  - `wecom_secret`
  - `wecom_agent_id`

**影响：**

- 发送器虽然会读取这些配置，但后台默认没有录入入口，部署后极易处于“代码支持、后台不可配”的状态。

#### 🔴 阻塞项 2：成员 ID 维护闭环不完整

- 服务人员编辑页已有 `wecom_userid` 输入项。  
  参考：`admin/src/views/staff/lists/edit.vue:66-68,687`
- 但顾问侧未看到同等明确的后台维护入口，而咨询分配逻辑已经会尝试给顾问发企微消息。  
  参考：`server/app/api/logic/CustomerServiceLogic.php:171`

**影响：**

- 业务侧可能触发了发送，但接收人缺少可用的企微成员 ID，最终仍无法投递。

### 5.3 审计判断

- 企业微信链路属于“发送器结构基本正确，但默认配置链路不完整”的状态。
- 当前仓库里能证明“会尝试发送”，但不能证明“默认就能成功发送”。

### 5.4 结论

- 企业微信内部消息：**当前不能认定为默认可跑通**。
- 必须补齐后台企微配置入口与接收人 `wecom_userid` 维护闭环，才具备真正落地条件。

## 6. 额外发现的流程缺口与 Bug

### 6.1 候补通知存在自动流与手动流不一致

#### 🟡 建议项：自动流会发“站内 + 订阅”，手动流只发站内

- `Waitlist` 模型自动处理时，会同时发订阅消息和站内消息。  
  参考：`server/app/common/model/schedule/Waitlist.php:190-258`
- 后台 `WaitlistLogic` 手动通知时，只调用了站内消息服务。  
  参考：`server/app/adminapi/logic/schedule/WaitlistLogic.php:164-169`

**影响：**

- 同一个“候补释放/候补提醒”业务，在不同入口触发时用户收到的消息渠道不一致。
- 从产品体验和排障角度看，这属于明显的不一致行为。

### 6.2 候补过期存在未执行到的死分支风险

#### 🟡 建议项：命令只跑预约日期过期，未跑 `expire_time` 过期

- `Waitlist` 模型内同时存在：
  - `processExpiredWaitlists()`  
    参考：`server/app/common/model/schedule/Waitlist.php:284`
  - `processPastDateWaitlists()`  
    参考：`server/app/common/model/schedule/Waitlist.php:323`
- 但实际命令 `expire_waitlists` 当前执行的是 `processPastDateWaitlists(100)`。  
  参考：`server/app/common/command/ExpireWaitlists.php:29`

**影响：**

- 如果业务原本希望按 `expire_time` 精确过期，当前命令不会覆盖这条分支。
- 这会导致代码里存在实现但不被当前调度使用的路径。

## 8. 建议整改顺序

### P0：先补“能否发得出去”的阻塞项

1. 为订阅消息补齐真实模板 ID，并把场景与模板绑定做成可配置、可校验闭环。
2. 在后台补齐企业微信配置项录入入口：`wecom_enabled`、`wecom_corp_id`、`wecom_secret`、`wecom_agent_id`。
3. 明确顾问与服务人员的 `wecom_userid` 维护路径，避免业务触发后没有有效接收人。

### P1：补齐前端授权与业务触发闭环

1. 将售后类订阅授权接入实际退款、工单相关页面。
2. 把退款审核完成、工单状态更新等业务事件与订阅发送器做显式绑定。

### P2：统一消息行为，减少入口差异

1. 统一候补自动流和手动流的消息渠道策略。
2. 统一候补过期任务的执行口径，明确到底以预约日期还是 `expire_time` 为准。

## 9. 最终判断

- 站内消息：**基本完善，当前可认为基本能跑通。**
- 微信订阅消息：**当前不完善，不能按默认状态认定为已跑通。**
- 企业微信内部消息：**当前不完善，不能按默认状态认定为已跑通。**
- 候补通知与过期处理：**存在流程不一致和死分支风险，建议尽快收口。**
